## Learnings

1. **Browser context code requires special handling**: `page.evaluate()` runs in browser where `document`/`window` exist but TypeScript doesn't know. Solutions:
   - Use `page.$$eval()` or `page.$eval()` for type-safe element queries
   - Use `@ts-expect-error` comments sparingly for unavoidable cases
   - Downgrade strict type rules to warnings for these patterns

2. **Async methods without await**: ESLint's `require-await` rule flags async methods that don't use await. Fix by either:
   - Remove `async` keyword and return `Promise.resolve()`
   - Or add actual async operations

3. **Detector registration pattern**: New detectors must be registered in 3 places:
   - `src/cli/commands/scan.ts` - CLI command
   - `src/mcp/server.ts` - MCP server
   - `src/determinism/replayer.ts` - HAR replay validation

## Post-Mortem

### What Worked
- Using `page.$$eval()` instead of `page.evaluate()` with document queries - provides type safety
- Creating separate `tsconfig.eslint.json` extending base config - cleanly fixes test file parsing
- Downgrading unsafe-any rules to warnings - pragmatic for browser context code

### What Failed
- Tried using `HTMLAnchorElement` type in `page.evaluate()` → Failed because DOM types not available in Node context
- Initial SEO detector used `page.evaluate()` with document → Fixed by switching to individual `page.$()` and `page.$$eval()` calls

### Key Decisions
- Decision: Downgrade strict type rules to warnings instead of errors
  - Alternatives: Add DOM lib to tsconfig, use extensive @ts-expect-error
  - Reason: Browser context code inevitably uses any; warnings preserve visibility without blocking builds
